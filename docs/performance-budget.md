# パフォーマンス予算ガイド

このドキュメントでは、本プロジェクトのパフォーマンス予算について説明します。パフォーマンス予算とは、Webサイトのパフォーマンスを維持するために設定する目標値と制限値のことです。

---

## パフォーマンス指標の説明

### Core Web Vitals（コアウェブバイタル）

Google が定義する、ユーザー体験を測定する3つの重要な指標です。

#### LCP (Largest Contentful Paint)

**説明**: ページの最大コンテンツ（通常はメイン画像やテキストブロック）が表示されるまでの時間を測定します。ユーザーが「このページは読み込まれた」と感じるまでの時間の指標です。

**目標値**: ≤ 2.5秒

**良い例**:
- 画像を適切に最適化している（WebP形式、適切なサイズ）
- フォントを効率的に読み込んでいる（font-display: swap）
- 重要なコンテンツを優先的に読み込んでいる

**悪い例**:
- 大きすぎる画像をそのまま使用している
- フォント読み込みでレンダリングがブロックされている
- サーバーの応答が遅い

#### INP (Interaction to Next Paint)

**説明**: ユーザーがページ上で操作（クリック、タップ、キー入力など）を行ってから、次の画面更新が表示されるまでの時間を測定します。ページの応答性の指標です。

**目標値**: ≤ 200ms

**良い例**:
- JavaScriptを非同期で読み込んでいる
- 重い処理を遅延実行している
- イベントハンドラーを効率的に実装している

**悪い例**:
- 大量のJavaScriptが同期的に実行されている
- 長時間実行されるタスクがメインスレッドをブロックしている
- イベントリスナーが最適化されていない

**注記**: INPは2024年9月にFID（First Input Delay）に代わって公式指標になりました。

#### CLS (Cumulative Layout Shift)

**説明**: ページ読み込み中に、予期せぬレイアウトのずれがどれだけ発生するかを測定します。視覚的な安定性の指標です。

**目標値**: ≤ 0.1

**良い例**:
- 画像に明示的な `width` と `height` 属性を指定している
- 広告や埋め込みコンテンツ用のスペースを事前に確保している
- Webフォント読み込み時のテキストのちらつきを防いでいる

**悪い例**:
- 画像サイズが指定されていない（読み込み後にレイアウトが変わる）
- 動的に挿入されるコンテンツがレイアウトを押し下げる
- CSSアニメーションで要素が大きく移動する

### Lighthouse スコア

Googleが提供する自動監査ツール「Lighthouse」の4つのカテゴリスコアです。

#### Performance（パフォーマンス）

**説明**: ページの読み込み速度と応答性を総合的に評価します。LCP、INP、CLSなどの指標を含みます。

**目標値**: ≥ 85点（100点満点）

**評価内容**:
- 初期表示速度
- インタラクティブになるまでの時間
- リソースの最適化度
- キャッシュ戦略

#### Accessibility（アクセシビリティ）

**説明**: すべてのユーザー（障害のある方を含む）がページを利用できるかを評価します。

**目標値**: ≥ 95点（100点満点）
**現在の暫定値**: ≥ 91点（UI改善完了後に95点へ引き上げ予定）

**評価内容**:
- 画像のalt属性
- カラーコントラスト比
- フォーカス可能な要素
- ARIAラベル

#### Best Practices（ベストプラクティス）

**説明**: Webのベストプラクティスに従っているかを評価します。

**目標値**: ≥ 90点（100点満点）

**評価内容**:
- HTTPSの使用
- コンソールエラーの有無
- 非推奨APIの使用
- セキュリティ設定

#### SEO（検索エンジン最適化）

**説明**: 検索エンジンがページを理解しやすいかを評価します。

**目標値**: ≥ 95点（100点満点）

**評価内容**:
- メタタグの設定
- クロール可能性
- モバイルフレンドリー
- 構造化データ

## 測定手順

### 1. Lighthouse（Chrome DevTools）

**最も手軽な測定方法**

1. Google Chromeでページを開く
2. 右クリック → 「検証」を選択
3. DevToolsの「Lighthouse」タブを選択
4. 「Performance」「Accessibility」「Best Practices」「SEO」にチェック
5. 「Analyze page load」をクリック

**メリット**:
- 開発環境ですぐに測定できる
- 詳細な改善提案が得られる
- 無料で使用できる

**注意点**:
- ローカル環境のパフォーマンスに依存する
- 一度の測定では変動がある（3回測定して中央値を取る）

### 2. PageSpeed Insights

**本番環境のリアルユーザーデータを確認できる**

1. [PageSpeed Insights](https://pagespeed.web.dev/) にアクセス
2. URLを入力して「分析」をクリック
3. 「フィールドデータ」と「ラボデータ」の両方を確認

**メリット**:
- 実際のユーザーのデータ（CrUX）が見られる
- 本番環境の測定が簡単
- モバイルとデスクトップの両方を測定できる

**注意点**:
- 公開されているURLのみ測定可能
- フィールドデータは28日分のデータ蓄積が必要

### 3. Lighthouse CI（自動測定）

**プロジェクトでは自動測定が設定済み**

GitHub Actionsで自動的に測定されます。

**トリガー**:
- `001-tech-blog` ブランチへのプッシュ
- `develop` または `001-tech-blog` ブランチへのプルリクエスト

**測定内容**:
- ホームページ、タグページ、カテゴリページ
- 各ページ3回測定して中央値を採用

**閾値**:
```json
{
  "Performance": 85点以上,
  "Accessibility": 91点以上,
  "Best Practices": 90点以上,
  "SEO": 95点以上
}
```

**結果の確認**:
- プルリクエストに自動でコメントが投稿される
- 閾値を下回るとCIが失敗する

### 4. WebPageTest（詳細分析）

**最も詳細な分析が可能**

1. [WebPageTest](https://www.webpagetest.org/) にアクセス
2. URLと測定条件を入力
3. 「Start Test」をクリック

**メリット**:
- ネットワーク速度を指定できる（3G、4G、光回線など）
- ウォーターフォールチャートで詳細な読み込み順序を確認できる
- ビデオで視覚的に確認できる

**使用タイミング**:
- 月1回の定期監査
- パフォーマンス問題の詳細調査

## バンドルサイズ予算

リソースサイズの制限値です。

| リソース | 予算 | 説明 |
|---------|------|------|
| **HTML** | ≤ 50KB | ページテンプレート全体 |
| **CSS** (gzip圧縮後) | ≤ 15KB | site.cssのみ |
| **JavaScript** (gzip圧縮後) | ≤ 10KB | GA4と最小限のスクリプト |
| **合計** | ≤ 150KB | ホームページの総サイズ |
| **画像** | ≤ 30KB | ホームページのヘッダー画像など |

**測定方法**:
```bash
npm run build
du -sh _site/
```

## 警告・エラーの基準値

### Yellow（警告）— 改善を検討

パフォーマンスは許容範囲内ですが、改善の余地があります。月1回の定期監査で対応を検討します。

| 項目 | 基準値 |
|------|-------|
| **LCP** | 2.5秒 - 4.0秒 |
| **INP** | 200ms - 500ms |
| **CLS** | 0.1 - 0.25 |
| **Performance** | 75点 - 85点 |
| **Accessibility** | 85点 - 95点 |

### Red（エラー）— 即時対応が必要

パフォーマンスが許容範囲を下回っています。CI/CDでビルドが失敗し、本番デプロイが防止されます。

| 項目 | 基準値 |
|------|-------|
| **LCP** | > 4.0秒 |
| **INP** | > 500ms |
| **CLS** | > 0.25 |
| **Performance** | < 75点 |
| **Accessibility** | < 85点 |

## パフォーマンス問題の改善方法

### LCPが遅い場合

**原因**:
- 画像サイズが大きすぎる
- フォント読み込みが遅い
- サーバー応答が遅い

**改善方法**:
1. 画像を最適化する（WebP形式、適切なサイズ）
2. `font-display: swap` を設定する
3. 重要なリソースをプリロードする
4. キャッシュを有効化する

**確認コマンド**:
```bash
npm run build
du -sh _site/images/
```

### INPが遅い場合

**原因**:
- JavaScriptの実行が重い
- イベントハンドラーが非効率

**改善方法**:
1. JavaScriptを非同期で読み込む
2. 不要なスクリプトを削除する
3. Google Analyticsを非同期モードで読み込む

### CLSが大きい場合

**原因**:
- 画像サイズが未指定
- 動的コンテンツの挿入

**改善方法**:
1. すべての画像に `width` と `height` を指定する
2. 動的コンテンツ用のスペースを事前に確保する
3. Webフォント読み込み時のレイアウトシフトを防ぐ

**確認コマンド**:
```bash
npm run test:alt-text  # 画像のalt属性とサイズ指定を確認
```

### Accessibilityスコアが低い場合

**原因**:
- 画像にalt属性がない
- カラーコントラスト比が低い
- キーボード操作できない要素がある

**改善方法**:
1. すべての画像に適切なalt属性を追加する
2. WebAIM Contrast Checkerで色のコントラストを確認する
3. インタラクティブな要素にキーボードフォーカスを追加する

**確認コマンド**:
```bash
npm run test:a11y        # アクセシビリティテスト
npm run test:alt-text    # 画像のalt属性確認
```

## ドキュメント更新フロー

### このドキュメントを更新するタイミング

1. **目標値の変更時**
   - パフォーマンス予算の基準を見直した場合
   - Core Web Vitalsの公式基準が更新された場合

2. **測定ツールの追加・変更時**
   - 新しい測定ツールを導入した場合
   - 測定手順が変更された場合

3. **新しい指標の追加時**
   - Googleが新しいCore Web Vitals指標を追加した場合
   - プロジェクト独自の指標を追加した場合

### 更新手順

1. このファイル（`docs/performance-budget.md`）を編集する
2. 変更内容を `.lighthouserc.json` にも反映する（閾値変更の場合）
3. `specs/001-tech-blog/performance-budget.md` の技術仕様も更新する
4. プルリクエストを作成して、チームでレビューする
5. マージ後、チームに変更内容を共有する

### 関連ファイル

- **このドキュメント**: `docs/performance-budget.md` — 実用的なガイド
- **技術仕様書**: `specs/001-tech-blog/performance-budget.md` — 詳細な技術仕様
- **CI設定**: `.lighthouserc.json` — Lighthouse CIの閾値設定

## 監視スケジュール

### 自動監視（CI/CD）

**頻度**: 毎回ビルド時
**ツール**: Lighthouse CI（GitHub Actions）
**アクション**: スコアが閾値を下回るとビルドが失敗

### 月次監査

**頻度**: 毎月第1月曜日
**ツール**: WebPageTest + Lighthouse（手動実行）
**レポート**: `specs/001-tech-blog/audit-logs/` に保存

### 四半期レビュー

**頻度**: 3ヶ月ごと（Q1/Q2/Q3/Q4）
**内容**: 長期トレンド分析、予算見直し
