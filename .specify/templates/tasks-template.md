# タスク一覧: [FEATURE NAME]

**入力**: `/specs/[###-feature-name]/` の設計ドキュメント
**前提条件**: plan.md（必須）、spec.md（ユーザーストーリーが必要）、research.md、data-model.md、contracts/

**テスト**: 下記の例にはテストタスクを含みます。テストは任意で、仕様書で明示されている場合のみ含めます。

**構成**: タスクはユーザーストーリーごとにグルーピングし、各ストーリーを独立実装・独立テストできるようにします。

## 形式: `[ID] [P?] [Story] 説明`

- **[P]**: 並列実行可能（異なるファイル、依存なし）
- **[Story]**: 対応するユーザーストーリー（例: US1, US2, US3）
- 説明には正確なファイルパスを含める

## パス規約

- **単一プロジェクト**: リポジトリルートの `src/`, `tests/`
- **Web アプリ**: `backend/src/`, `frontend/src/`
- **モバイル**: `api/src/`, `ios/src/` または `android/src/`
- 下記のパスは単一プロジェクトを想定。plan.md の構成に合わせて調整

## 依存関係と実行順序

### フェーズ依存

- **セットアップ（フェーズ1）**: 依存なし、すぐ着手可能
- **基盤（フェーズ2）**: セットアップ完了が必須、全ストーリーをブロック
- **ユーザーストーリー（フェーズ3+）**: 基盤完了が必須
  - 体制があれば並行実装可能
  - または優先度順（P1 → P2 → P3）で実施
- **仕上げ（最終フェーズ）**: すべての必要なユーザーストーリー完了が必須

### ユーザーストーリー依存

- **ユーザーストーリー1（P1）**: 基盤完了後に着手可能、他ストーリーへの依存なし
- **ユーザーストーリー2（P2）**: 基盤完了後に着手可能、US1 と連携する場合は独立性を維持
- **ユーザーストーリー3（P3）**: 基盤完了後に着手可能、US1/US2 と連携する場合は独立性を維持

### 各ユーザーストーリー内の順序

- テスト（含める場合）は実装前に作成し、失敗を確認する
- モデル → サービス → エンドポイントの順
- 主要実装 → 統合の順
- ストーリー完了後に次の優先度へ移行

### 並行実行の機会

- [P] のセットアップタスクは並行実行可能
- [P] の基盤タスクはフェーズ2内で並行実行可能
- 基盤完了後、ユーザーストーリーは並行実装可能
- [P] のテストは並行実行可能
- 同一ストーリー内の [P] モデルは並行実行可能
- 異なるストーリーは担当者を分けて並行実行可能

## 実装戦略

### MVP 優先（ユーザーストーリー1のみ）

1. フェーズ1: セットアップ完了
2. フェーズ2: 基盤完了（重要）
3. フェーズ3: ユーザーストーリー1完了
4. **停止して検証**: ユーザーストーリー1を単独テスト
5. 必要ならデプロイ/デモ

### 段階的デリバリー

1. セットアップ + 基盤完了 → 基盤準備完了
2. ユーザーストーリー1追加 → 単独テスト → デプロイ/デモ（MVP）
3. ユーザーストーリー2追加 → 単独テスト → デプロイ/デモ
4. ユーザーストーリー3追加 → 単独テスト → デプロイ/デモ
5. 各ストーリーは既存価値を壊さずに積み上げる

### 並行チーム戦略

複数人開発の場合:

1. チームでセットアップ + 基盤を完了
2. 基盤完了後:
   - 開発者A: ユーザーストーリー1
   - 開発者B: ユーザーストーリー2
   - 開発者C: ユーザーストーリー3
3. 各ストーリーが独立完了して統合

## 注意事項

- [P] タスク = 異なるファイル、依存なし
- [Story] ラベルで対応ストーリーを追跡
- 各ストーリーは独立して完結・テスト可能
- 実装前にテスト失敗を確認
- タスクまたはタスク群の区切りでコミット
- 各チェックポイントで単独検証
- 回避: 曖昧タスク、同一ファイル競合、独立性を壊すクロス依存
