# Pull Request ガイドライン

Pull Request を作成・レビューする際のガイドラインです。

---

## PR 作成時のポイント

### 1. ブランチ名は明確に
変更内容が一目でわかる名前を付けてください。

**良い例：**
- `feature/add-tag`
- `feature/seo-optimize`
- `fix/search-bug`

**悪い例：**
- `feature/update`
- `fix/bug`

---

### 2. 説明を記載
PR の説明には、以下を含めてください：

- **なぜこの変更が必要なのか** — 背景・理由
- **何を変更したか** — 変更内容の概要
- **テストについて** — 動作確認の方法、追加したテスト
- **関連 Issue** — `Fixes #123` など

**PR 説明テンプレート例：**
```markdown
## 概要
新しいタグ管理機能を実装しました。

## 背景
#123 のユーザー要求に応じて、記事に複数のタグを付与できるようにしました。

## 変更内容
- `Tag` モデルを新規作成
- 記事とタグのリレーションシップを設定
- タグ管理 UI を実装

## テスト
- ローカル環境で動作確認
- 新規テストケースを追加（`test/tag.test.js`）

## 関連 Issue
Fixes #123
```

---

### 3. 小粒度の PR を心がける
複数機能の混在は避けてください。

**良い例：**
- PR 1: タグ機能の実装
- PR 2: タグの削除機能

**悪い例：**
- PR: タグ機能 + 検索機能 + UI 全体リファクタリング

---

### 4. テストを追加
テスト可能な機能にはテストコードを含めてください。

---

## PR レビューのポイント

### レビュー時のチェック項目

- [ ] コミットメッセージは規約に従っているか
- [ ] コード品質は適切か
- [ ] テストは十分か
- [ ] ドキュメント更新は必要か
- [ ] 不要なコミットは含まれていないか

---

## マージ方針

- **必ずプルリクエスト経由でマージ** — main と develop への直接プッシュは禁止
- **--no-ff フラグを使用** — ブランチ履歴を保持（release / hotfix の場合）
- **コンフリクト解決** — マージ前に必ず解決
